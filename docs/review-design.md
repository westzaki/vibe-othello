# Review Design

## Purpose

このドキュメントは、Vibe Othello の Review 体験設計を定義する。

Review は、対局後にプレイヤーを責めるための分析機能ではない。
「負けた理由」を細かく並べるのではなく、次に勝てるポイントを一緒に見つけるための学習体験である。

Review の文章は分析レポートではなく、プレイヤーが盤面をもう一度見られるようにする短いコーチングコピーとして扱う。
特定の話し手がいる前提にはせず、画面全体の案内として自然に読めることを重視する。

Review が目指す感情の変化は、次の一文に集約される。

> 「あと一手で勝てたのに」を、「次は勝てる」に変える。

そのため、Review 関連の実装や UI 判断では、正確な解析量や情報量よりも、プレイヤーが次の対局や練習で試したくなることを優先する。

## Relationship to Product Documents

Review の設計判断は、次のドキュメントに従う。

- `docs/product-vision.md`
- `docs/learning-design.md`
- `docs/coach-persona.md`
- `docs/architecture.md`

Review は Product Vision の「次は勝てそう」という期待、Learning Design の「相手にいい手をあげない」を考えられるようになる方針、Coaching Voice のやさしく押し付けない声かけを実現する場所である。

実装上は、Review と Practice の境界を `docs/architecture.md` に合わせる。
Review は完了した対局の履歴を読み、Practice は選んだ局面の盤面スナップショットから別セッションとして開始する。
Review が完了済みの対局を壊したり、直接 Practice の状態を所有したりしない。

## Review Goals

- 負けた理由を責めるのではなく、次に勝てるポイントを見つける。
- プレイヤーが「ここを見ればよかったんだ」と理解できるようにする。
- すべてを説明しきるのではなく、次の一局で意識できる量に絞る。
- 良い判断を見つけて、負けた試合でも前向きに終われるようにする。
- 指摘された局面からすぐ練習できる導線を用意する。

## Basic Structure

Review は、毎回すべての手を並べるのではなく、次の3つの枠で構成する。

1. 今日のナイス
2. ここが分かれ道だったかも
3. 次はこれを試してみよう

### 1. 今日のナイス

プレイヤーができていた良い判断を扱う。

勝った試合だけでなく、負けた試合でも必ず良い判断を探す。
目的は、プレイヤーに「自分にも勝ちにつながる手が打てていた」と感じてもらうことである。

優先して褒めるもの：

- 角を取れた手
- 相手に角を渡さない形を選べた手
- 相手の合法手数を減らせた手
- たくさん取るより次の形を大事にできた手
- 終盤で石数や置ける場所を意識できた手

### 2. ここが分かれ道だったかも

勝敗や流れに影響した局面を、責めずに扱う。

「悪手」「ミス」と断定するのではなく、別の見方ができた場面として提示する。
プレイヤーが自分で考え直せるように、強い言い切りよりも「ここ、少し分かれ道だったかも」のような表現を優先する。

### 3. 次はこれを試してみよう

次の対局や練習で1つだけ試せる行動に落とし込む。

Review の最後は、抽象的な反省ではなく、次に使える小さな作戦で終える。
たとえば「角の近くに置く前に、相手が角を取れるか見てみよう」「たくさん返せる手のあと、相手の置ける場所を数えてみよう」のように、盤面で確認できる行動にする。

## Position Priority

Review で優先する局面は次の通り。

- 角を取れた局面
- 角を逃した局面
- 相手に角を渡した局面
- 空き角の近くなど、危険マスに置いた局面
- 相手の合法手数が増えた局面
- 相手の合法手数が減った局面
- 評価の流れが大きく変わった局面
- たくさん返せる手を選んだ結果、次の形が苦しくなった局面
- 終盤で石数や残り手数が勝敗に影響した局面

局面選定では、解析上の差が大きいだけでは不十分である。
プレイヤーが次の対局で見つけやすく、再挑戦しやすく、学習テーマにつながる局面を優先する。

## Display Policy

Review の表示は、情報量を増やすことよりも、理解しやすさと前向きさを優先する。

- 全手解析ではなく、重要局面に絞る。
- 評価値、探索深度、候補手順位などの数値を前面に出さない。
- 子どもでも分かる言葉にする。
- 一度に読む文章量を短くする。
- 専門用語を使う場合は、盤面上の行動とセットにする。
- 「悪手」「ミス」「ダメ」「間違い」は避ける。
- プレイヤーの判断を否定せず、次の見方を提案する。

Review は、プレイヤーが自分で考えた感覚を残す。
そのため、常に最善手を断定的に教える必要はない。
具体的な候補を示す場合でも、「こっちも試してみる？」のように練習へつながる言い方を優先する。

## Tone and Copy

Review の声かけは、`docs/coach-persona.md` の Coaching Voice に従う。
特定の話し手らしさではなく、短く、前向きで、盤面に戻りやすい案内であることを重視する。

基本方針：

- やさしく、落ち着いている。
- 命令しない。
- 褒めるときは、結果だけでなく考え方を褒める。
- 指摘するときは、責めずに次の一手へつなげる。
- 「正解を教える先生」ではなく、「次に見る場所を一緒に絞る案内」として話す。
- 1P対局では、勝ち負けを引きずりすぎず、次の行動へつなげる。
- 2P対局では、どちらか一方だけを責めたり持ち上げたりしない。

使いやすい表現：

- 「ここ、いい手だったと思う」
- 「ここ、分かれ道だったかも」
- 「次はこのへんを見てみよう」
- 「相手の置ける場所、少し減らせていたね」
- 「角の近く、もう一回だけ見てみる？」
- 「この手もかなりいい判断だったと思う」

避ける表現：

- 「これは悪手です」
- 「ミスです」
- 「ダメな手です」
- 「正解はここです」
- 「なぜこれを選んだのですか」

## Practice Flow

Review は、説明を読んで終わる画面ではない。
指摘された局面からすぐ再挑戦できることが、Review 体験の一部である。

練習導線で重視すること：

- 指摘された局面からすぐ再挑戦できる。
- 元の対局結果を壊さない。
- Review 画面に戻れる。
- 別の手を試せる。
- 練習後に改善できた点を短く褒める。

練習後のフィードバックは、長い採点ではなく、1つの改善を見つけて伝える。
たとえば「今度は相手の角チャンスを減らせたね」のように、Review で見たポイントと練習結果をつなげる。

## Implementation Guidelines

Review 関連の変更では、次の判断基準を使う。

- その変更は、プレイヤーが次に勝てるポイントを見つける助けになるか。
- その変更は、情報過多にならず重要局面に絞れているか。
- その変更は、子どもでも読める言葉と量になっているか。
- その変更は、プレイヤーを責めずに次の行動へつなげているか。
- その変更は、Practice へ自然につながるか。
- その変更は、Review と Practice のアーキテクチャ境界を守っているか。

これらを満たさない Review 機能は、実装前に小さく分けるか、目的を見直す。

## Non-goals

Review の主目的ではないもの：

- 完全解析
- 難しい定石解説
- 評価値中心の UI
- 全手の細かいレビュー
- 上級者向けの理論最善手解説
- 探索深度や候補手スコアを見せる解析画面
- プレイヤーの負け方を細かく採点する体験

これらは将来的に上級者向けモードとして扱う可能性はあるが、Vibe Othello の基本 Review 体験には含めない。
